# WooYun Legacy

> 22,132 个真实业务逻辑漏洞，教会 AI 像资深安全专家一样思考

## 这是什么

这是一个 **Claude Code Skill**，基于 WooYun（2010-2016）收录的 22,132 个业务逻辑漏洞案例，提炼出覆盖 **6 大领域、33 类漏洞** 的 Web 应用安全测试方法论。

装上这个 Skill 后，Claude 不再是泛泛而谈的安全顾问——它能引用真实公司案例、给出量化统计、按 WooYun 独有的「任意X」分类法输出精准的测试方案。

### 与通用 AI 的区别

| 维度 | 无 Skill 的 Claude | 装了 WooYun Legacy |
|------|-------------------|-------------------|
| 案例引用 | 通用安全建议，无真实案例 | 引用具体公司名 + 漏洞描述 |
| 数据支撑 | 主观判断（"风险较高"） | 量化统计（"1,056 案例，68.7% 高危"） |
| 攻击分类 | 常规 OWASP 分类 | WooYun 独有分类（任意账号/操作/查看/修改/删除） |
| 测试深度 | 3-5 种攻击模式 | 每个领域 10+ 种攻击模式 + 绕过技巧 |

## 评估基准

我们对 Skill 进行了 12 组全领域对照评测（with_skill vs without_skill），覆盖全部 6 个领域和 33 类漏洞：

| 指标 | With Skill | Without Skill | 差异 |
|------|-----------|---------------|------|
| **断言通过率** | 100% (72/72) | 64.6% (47/72) | **+54.8%** |
| 平均耗时 | 222s | 164s | +35% |
| 平均 Token | 41K | 28K | +46% |

**关键发现：**
- WooYun 案例引用断言：with_skill **12/12 通过**，without_skill **0/12 通过**
- WooYun 统计数据断言：with_skill **12/12 通过**，without_skill **0/12 通过**
- 领域专业断言（攻击模式、测试步骤等）：without_skill 也能通过 98%——说明 Skill 的核心价值在于注入**真实数据**而非通用知识

### 场景一：电商支付安全测试

> **你说：** 我在做一个电商平台的安全审计，这个平台有购物车、在线支付（支付宝/微信）、订单管理、退款流程。请帮我制定一个针对支付和订单流程的安全测试方案。

<details>
<summary><b>点击展开 Claude 的回复（节选）</b></summary>

Claude 会输出一份结构化的测试方案，包含：

**订单状态机非法转移检测（WooYun 订单篡改 1,227 案例，74.2% 高危）：**

| 非法转移 | 攻击意图 | 风险等级 |
|---------|---------|---------|
| 待支付 → 已支付 | 跳过支付环节 | 严重 |
| 待支付 → 已完成 | 完全绕过支付和物流 | 严重 |
| 已发货 → 已退款 | 收货后退款（白嫖） | 高 |
| 已完成 → 退款中 | 已确认收货后仍触发退款 | 高 |
| 已退款 → 已退款 | 重复退款 | 严重 |

**每个测试点都有具体步骤和真实案例引用：**

> **WooYun 模式：** "M1905电影网价值2588套餐只要5毛钱" — 客户端提交价格被服务器直接信任。
>
> **测试步骤：**
> 1. 正常下单（如总额 ¥299），进入支付页面
> 2. 拦截"发起支付"的请求
> 3. 修改请求中的金额参数：`amount=0.01` / `amount=0` / `amount=-1`
> 4. 观察：支付宝/微信支付页面显示的金额是否为篡改后的值？

**附录包含量化统计：**

| 子类别 | 案例数 | 高危占比 |
|--------|--------|----------|
| 支付绕过 | 1,056 | 68.7% |
| 订单篡改 | 1,227 | 74.2% |
| 金额篡改 | 176 | 83.0% |
| 价格篡改 | 70 | 74.3% |

> **你说：** 我们公司有一个多租户 SaaS 平台，提供 REST API。我需要你帮我设计一个针对越权访问的完整测试计划，包括水平越权（IDOR）、垂直越权、未授权访问。

<details>
<summary><b>点击展开 Claude 的回复（节选）</b></summary>

Claude 会基于授权域 6,838 个案例生成测试计划，包含：

**WooYun 独有的「任意X」分类法（529 案例）：**

| 子类别 | 案例数 | 高危占比 | 测试要点 |
|--------|--------|----------|---------|
| 任意账号操作 | 220 | 86.4% | 修改他人手机号/邮箱/密码 |
| 任意查看 | 145 | 62.1% | 遍历他人订单/信息/记录 |

| 任意删除 | 35 | 71.4% | 删除他人数据/文件 |
| 任意操作 | 40 | 72.5% | 代替他人执行敏感操作 |

**IDOR 测试矩阵（不只是"改 ID"）：**

| 技术 | 说明 | 示例 |
|------|------|------|
| 直接 ID 替换 | 最基础，约占 30% | `user_id=1001` → `user_id=1002` |
| ID 编码绕过 | Base64/Hex 编码的 ID | 解码 → 修改 → 重新编码 |
| 参数污染 | 同名参数覆盖 | `?id=me&id=1002` |
| JSON 嵌套篡改 | body 中的嵌套对象 | `{"user": {"id": 1002}}` |
| 批量操作接口 | 数组中混入他人 ID | `{"ids": [1001, 1002, 1003]}` |
| GraphQL 越权 | 查询中替换节点 ID | `node(id: "other-user-id")` |

**统计支撑：** 58.2% 的未授权访问发现 = 管理后台完全暴露在互联网上，无任何身份认证。

### 场景三：代码安全审计

> **你说：** 帮我审计这段电商下单接口代码，找出所有业务逻辑漏洞。

<details>
<summary><b>点击展开 Claude 的回复（节选）</b></summary>

Claude 会按四阶段方法论进行审计，输出结构化报告：

```
漏洞 1：金额可篡改为负数 — 优惠券折扣无上限
├── 严重级别：Critical
├── 领域：金融 — 金额篡改（176 案例，83.0% 高危）
├── 代码定位：total_price -= coupon.discount_amount（无下限校验）
├── 业务影响：攻击者可构造 total_price 为负数，实现"倒贴"
└── 修复代码：total_price = max(total_price, 0.01)

漏洞 2：优惠券无归属校验 — 任意用户可复用
├── 严重级别：High
├── 领域：授权 — 任意操作（40 案例，72.5% 高危）
├── 代码定位：仅查询 coupon_code 有效性，未校验 user_id 归属
├── 业务影响：用户 A 的专属优惠券可被用户 B 使用
└── 修复代码：增加 AND user_id = current_user.id 条件

漏洞 3：数量参数无范围校验 — 负数/零/超大值
├── ...（共发现 9 个漏洞）
```

每个漏洞都附带 WooYun 真实案例和统计数据作为风险佐证。

</details>

### 场景五：竞态条件专项测试

> **你说：** 我在测试一个在线票务系统，怀疑存在竞态条件漏洞。系统有抢票、支付、退票、积分兑换等功能。帮我设计一个竞态条件专项测试方案，需要包含测试脚本。

<details>
<summary><b>点击展开 Claude 的回复（节选）</b></summary>

Claude 会输出包含 Python 并发脚本的测试方案：

```python
# 并发抢票竞态测试
import asyncio
import aiohttp

async def race_buy_ticket(session, url, ticket_id, token):
    payload = {"ticket_id": ticket_id, "quantity": 1}
    headers = {"Authorization": f"Bearer {token}"}
    async with session.post(url, json=payload, headers=headers) as resp:
        return await resp.json()

async def main():
    url = "https://target.com/api/ticket/buy"
    async with aiohttp.ClientSession() as session:
        # 50 个并发请求抢同一张票
        tasks = [race_buy_ticket(session, url, "TICKET-001", token)
                 for _ in range(50)]
        results = await asyncio.gather(*tasks)
        success = [r for r in results if r.get("status") == "success"]
        print(f"成功购买: {len(success)} 次（预期: 1 次）")
```

还包含状态机非法转换测试、TOCTOU 竞态检测等，每个都有 WooYun 案例佐证。

</details>

## 方法论核心

### 四阶段方法

```
第一阶段：业务流程映射
  └─ 理解业务 → 映射用户旅程 → 识别状态转移 → 映射信任边界

第二阶段：假设形成
  └─ 基于 6 大领域参考文件，形成"[流程X] 可能受到 [攻击Y]"的假设

第三阶段：针对性手动测试
  └─ 准备多角色账号 → 拦截请求 → 单参数修改 → 观察服务器行为

第四阶段：影响评估与文档化
  └─ 证明业务影响 → 匹配 WooYun 模式 → 输出可操作的修复建议
```

### 运营商渗透

> **提问：** 请你思考告诉我运营商有哪些常见的漏洞？有哪些不常见的攻击面，如何实现 getshell?

<details>
<summary><b>点击展开完整回复</b></summary>

#### 一、运营商常见漏洞分类

##### 1. 弱口令问题（最高频）

| 系统类型 | 常见默认凭证 | 风险等级 |
|---------|------------|---------|
| 网络设备 | admin/admin, root/root | 高危 |
| 业务系统后台 | admin/123456, test/test | 高危 |
| 数据库 | sa/空密码, root/root | 高危 |
| BOSS 系统 | 工号/工号, 工号/123456 | 高危 |

##### 2. 越权漏洞（业务逻辑层）

```
典型场景：
├── 水平越权：查看/修改他人套餐、话费、通话记录
├── 垂直越权：普通用户 → 客服权限 → 管理员权限
└── 接口越权：手机号遍历获取用户信息
```

#### 二、不常见但高价值的攻击面

- 增值业务平台（SP/CP 合作平台 → 短信网关 → BOSS 核心）
- 网络管理系统（华为 U2000/M2000、动环监控）
- 物联网卡管理平台（IoT 设备管理、M2M 平台）
- 内部 OA/ERP 系统（员工自助门户、VPN 接入点）

#### 三、GetShell 路径

```
路径一：Web 应用层突破
  Struts2 RCE → WebLogic 反序列化 → Shiro 反序列化 → 文件上传绕过

路径二：边界设备突破
  VPN 漏洞（Pulse Secure/Fortinet/深信服）→ 内网横向

路径三：供应链攻击
  第三方外包公司 → 开发测试环境 → 正式环境
```

</details>
